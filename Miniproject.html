Logistic Regression¶

Logistic regression uses any one of multiple possible solvers to find a logistic curve of best fit for the given data. This logistic curve is generated by the formula 1/(1+e^-(TX)), where is a vector of weights applied to each feature of an example within data matrix X (with examples m and features n+1). By altering these weights, the linear combination in the exponent of the function can be changed so that its cost function (measuring how inaccurate the regression was) can be minimized.

Logistic regression is adaptable and a good all-around utility. It runs fairly fast, and can usually be reduced to a closed-form linear-algebra-based solution, reducing the overall process time. By including features which are polynomial mutations of other features, sophisticated nonlinear decision boundaries can be formed. This does, however, leave the algorithm open to overfitting, therefore requiring regularization.

The MNIST dataset was loaded from openml.org, then randomly divided into training and testing datasets of 60,000 and 10,000 examples respectively, then standardized for optimal processing efficiency in the logistic regression algorithm. This random partition also means that accuracy varies slightly between runs of the same code.

This constant determines how much the cost function should weight the accuracy of the regression (a high emphasis on which can result in overfitting) relative to the regularization penalty (a high emphasis on which can result in underfitting). It is odd that this dataset seemed so resolutely immune to overfitting, its accuracy only improving as the C value went up even to the point of fantastically large C values, but was prone to underfitting, with low C values producing visible drops in accuracy. Especially odd was the reaction of the -matrix plots (generated using code from scikit-learn.org) to this change in C value. Since the features used in the regression were the individual pixels themselves, it is possible to plot the weights given to those features in the places of the pixels to produce a visual representation of the -matrix.

This at first seemed counterintuitive to me, since a perfect fit of the digit looks like an overfit while a more uniform mess looks like an underfit, but I was thinking about it the wrong way. These pretty pictures which look exactly like their digits are actually showing extremely high bias. The smooth shapes shown in the plot only show the major features of the digits and generalize too much. The pixelated disaster of the high-C plot is actually showing the high variance of an extremely fitted model, and the fact that this didnt affect the accuracy even when checking against a test dataset of 10,000 images suggests that the variation in the images from the original dataset really is extremely high and a high-variance model is appropriate.

Support Vector Machines¶

Support vector machines use some similar principles to logistic regression models, such as a cost function which needs to be minimized and a decision boundary generated at the end. However, they trim a lot of computational fat and make use of a lot of tricks to cut down execution time. When their C-values are set high, they try to create a decision boundary with as much of a data-free margin on either side of it as possible. Since they are so computationally efficient compared to logistic regression, they are good candidates for applying the otherwise computation-heavy concept of kernel functions. When using kernel functions, instead of using the variables in the data as features, the algorithm uses the outputs of a kernel function defining the similarity of each example to each other example as features. By changing the type and bandwidth of this kernel function, different decision boundaries can be achieved.

The MNIST dataset was randomly divided once again into training and testing datasets, but this time of 10,000 and 10,000 examples respectively. This random partition also means that accuracy varies slightly between runs of the same code.

Hyperplanes and Computer Vision¶

In the end, though they get there different ways, Logistic Regression and Support Vector Machines both look to create one or more hyperplanes which can separate the data points into one class or another. If you think of these separations as creating geometrically bounded spaces on an n-dimensional graph (where n is the number of features), what you get is a region assigned to each class. Contiguous regions on a cartesian graph, no matter the dimensionality, designate a relationship (or set of relationships) between the variables involved which must be true in order for an example to fall into the region. This is important when it comes to computer vision problems. The shape, and therefore meaning, of a picture is not determined by individual pixels independent of one another, it is determined by the relationships between pixels. Since decision-boundary-based methods inherently take relationships between features into account, they are well-suited to this task.

Naïve Bayes¶

Nave Bayes classifiers, unlike the other two, explicitly assume the opposite of the idea explained above. The basis of nave Bayes is Bayes theorem, and the assumption of the classification method is that features contribute to the probability of the example belonging to one class or another independently of one another. This is obviously not universally applicable, hence the label nave, but it works surprisingly often anyway and is computationally very fast owing to its vast simplicity relative to other methods.

That being said, the built-in independence assumption would seem to make nave Bayes unsuitable to computer vision, but as is often the case with nave Bayes, it surprisingly works in places where it seems like it shouldnt. Scikit-learn offers 5 different implementations of nave Bayes, each optimized for a different distribution in the data. I wasn't sure which one would be most applicable, so I tried all 5.

The Bernoulli and multinomial classifiers tended to come out as equally accurate, which is puzzling. MultinomialNB seems designed for this kind of data, but BernoulliNB is not. For some reason, binarizing the data does not seem to affect the accuracy. One possibility is that even though there may be shades of gray at the edges of lines, each pixel is basically 'light' or 'dark', and that the multinomial distribution of all pixels across all images is in fact bimodal, peaking at 0 and 255. By turning shades of gray into black or white, edges lose their anti-aliased fuzziness but the shape of the digit barely changes. This could be advantageous, since the loss of accuracy is minimal and the problem becomes much simpler (and therefore faster) to solve.
